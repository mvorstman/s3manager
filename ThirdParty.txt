================================================================================
  PROJECT OVERVIEW
================================================================================

S3Manager is a Go CLI tool for managing S3-compatible object storage.
The PowerShell companion (S3-Manager-v2.5.ps1) was retired on 2026-04-12.

  Go CLI (main.go + internal/)
     - Flag-based CLI; connects to any S3-compatible endpoint
     - Credentials via flags or env vars (S3_ENDPOINT, AWS_ACCESS_KEY_ID,
       AWS_SECRET_ACCESS_KEY, AWS_REGION)
     - Actions: list, upload, upload-folder, download, download-prefix, head, delete
     - Concurrency: configurable worker pool for upload-folder, download-prefix, delete
     - Retry: configurable max-attempts + backoff via AWS SDK v2

  Architecture (Go):
    internal/s3/       — low-level S3 primitives (client, uploader, downloader,
                         lister, head). Each file is a focused, testable unit.
    internal/engine/   — orchestration: worker pools, pagination loops, batch
                         delete, folder walk, result aggregation.
    main.go            — CLI entry point; flag parsing, env fallback, dispatch.

Priority: MEDIUM
----------------
[3] internal/s3/lister.go — the inner for-loop (lines 43-47) is de-indented
    relative to the surrounding code. Cosmetic but inconsistent.
    Fix: re-indent the loop body to match the outer block.

[4] internal/engine/delete.go — the verbose fmt.Printf block (lines 192-193)
    inside the live-delete loop has inconsistent indentation.
    Fix: align with the surrounding if-block.

[5] internal/s3/lister.go prints directly to stdout (side effects in the s3
    package layer). All other s3-layer functions return data and let callers
    print via PrintXxx(). Lister is the exception — makes it hard to test.
    Fix: return an []ObjectInfo slice; move printing to engine/list.go.

[6] uploader.go does not set ContentType on PutObjectInput.
    All objects land in S3 as application/octet-stream.
    Fix: detect MIME type from extension (mime.TypeByExtension) or accept it
    as an optional parameter.

Priority: LOW / FUTURE
-----------------------
[7] upload_jobs.go uses filepath.Walk; prefer filepath.WalkDir (Go 1.16+)
    to avoid extra Lstat calls per file. No functional impact, just cleaner.

[8] UploadFolder does not cancel on per-file errors (unlike delete/download).
    Currently all files are attempted regardless of prior failures.
    Consider adding a --fail-fast flag or at least a ctx cancel on first error.

[9] No tests. The package structure (separate s3 vs engine layers) makes
    testing straightforward — engine functions can be tested with a mock client.
    Suggested starting point: table-driven tests for BuildUploadJobs and
    localPathForObject (both are pure functions with no S3 dependency).
