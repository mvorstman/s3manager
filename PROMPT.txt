S3Manager Project – Resume Context

I am working on a Go-based CLI tool called S3Manager.

Architecture:
- main -> CLI / argument parsing / user-facing output
- engine -> orchestration, worker pool logic, aggregation
- internal/s3 -> low-level S3 operations

Project goals:
- Build a clean, professional S3-compatible CLI tool
- Learn Go properly while building something real
- Keep architecture clean and easy to extend
- Prefer step-by-step development over large rewrites

Current status:
- upload-folder is implemented using a worker pool
- discovery and execution are separated
- throughput reporting was added to the summary output
- worker count is configurable with --workers
- real-world performance testing was done successfully
- current upload-folder implementation is working correctly
- upload pipeline now includes retry, context-aware backoff, retry filtering, progress reporting, and retry visibility

Current upload architecture:
- internal/engine/upload_types.go
  - UploadJob
  - UploadFileResult
  - UploadFolderResult
- internal/engine/upload_jobs.go
  - BuildUploadJobs(...)
- internal/engine/upload.go
  - uploadWorker(...)
  - UploadFolder(...)
  - PrintUploadFolderResult(...)
- internal/s3/uploader.go
  - UploadFile(...)

Current design rules:
- Workers do NOT print
- Workers do NOT aggregate
- Engine aggregates results
- S3 layer only performs low-level S3 work
- Keep separation between discovery, execution, and presentation
- Continue step-by-step unless explicitly asked for a larger jump

Environment setup:
source ~/.s3manager-env

Known test dataset:
- Folder: ~/s3manager-test/temp-8155-uploadset
- Approximately 2000 files
- Approximately 7.6 MiB uploaded data
- Nested folder structure

Typical test command:
go run . \
  --action upload-folder \
  --bucket temp-8155 \
  --folder ~/s3manager-test/temp-8155-uploadset \
  --workers 40 \
  --prefix "TF-40/"

Observed benchmark results:
- workers=1   -> ~92s
- workers=8   -> ~9.5s
- workers=16  -> ~7.33s
- workers=32  -> ~2.95s
- workers=40  -> ~2.15s, throughput ~3.41 MiB/s
- workers=50  -> ~1.84s to ~1.94s
- workers=64  -> ~3.73s
- workers=100 -> ~2.43s
- workers=200 -> ~2.79s

Benchmark conclusion:
- Upload workload is I/O bound
- Parallelism improved performance dramatically
- Current sweet spot appears to be roughly 40-60 workers for this test set and environment
- More workers is not always better
